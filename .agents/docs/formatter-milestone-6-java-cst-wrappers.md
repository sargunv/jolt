# Formatter Milestone 6: Java CST Wrappers

## Goal

Build a production-quality typed Java CST wrapper layer on top of the existing
Java red tree before starting Java formatter layout work.

The wrapper layer is the public Java syntax API. Formatter code should consume
typed CST wrappers, not raw `SyntaxNode<JavaLanguage>` traversal. Raw red nodes
remain an internal implementation detail of `jolt_java_syntax`.

This milestone completes the Java syntax access layer. It is not a
formatter-critical subset and should not be scoped around whichever syntax the
first layout builder happens to touch.

## Non-Goals

- Do not build a semantic AST.
- Do not expose raw Java syntax nodes as a public escape hatch.
- Do not start Java layout-building work against incomplete wrappers.
- Do not introduce checked-in generated source files unless macro-based
  generation proves inadequate.
- Do not add recovery-specific wrapper APIs for partial formatting yet.

## Design

The existing green/red syntax infrastructure remains unchanged:

- green nodes and tokens provide immutable lossless storage,
- red nodes and tokens provide parent-aware traversal, offsets, and ranges,
- Java CST wrappers provide language-specific typed access over red nodes.

The wrapper layer should be declared from a single `macro_rules!` schema. The
schema is handwritten because Java grammar grouping is a language-design
decision, but the repetitive wrapper implementation should be generated by the
macro.

Keep the schema intentionally simple enough that `macro_rules!` is a good fit:
flat node lists, explicit grammar-family enum lists, and uniform generated
impls. Do not make the schema depend on inferred grammar relationships, nested
validation logic, or metadata processing that would turn the macro into a
parser. If the desired wrapper layer starts needing that complexity, redesign
the schema before reaching for a proc macro.

External source-level code generation is not part of this milestone.

## Public API Shape

The public API should expose typed wrapper structs and grammar-family enums.
Each wrapper owns or clones the underlying red node internally, but does not
expose it publicly.

Use an internal `nodes` module for the wrapper implementation. Do not expose a
public `cst` module path. `CST` is useful architecture language, but public
names should read as Java syntax nodes and syntax families, preferably
re-exported from the `jolt_java_syntax` crate root.

Example shape:

```rust
pub struct CompilationUnit {
    syntax: JavaSyntaxNode,
}

pub struct ImportDeclaration {
    syntax: JavaSyntaxNode,
}

pub enum TypeDeclaration {
    Class(ClassDeclaration),
    Record(RecordDeclaration),
    Enum(EnumDeclaration),
    Interface(InterfaceDeclaration),
    AnnotationInterface(AnnotationInterfaceDeclaration),
}
```

Public methods should expose stable syntax facts:

```rust
impl CompilationUnit {
    pub fn package_declaration(&self) -> Option<PackageDeclaration>;
    pub fn imports(&self) -> impl Iterator<Item = ImportDeclaration> + '_;
    pub fn type_declarations(&self) -> impl Iterator<Item = TypeDeclaration> + '_;
    pub fn text_range(&self) -> TextRange;
}
```

Internal methods may expose raw nodes to implement wrappers:

```rust
impl CompilationUnit {
    pub(crate) fn cast(node: JavaSyntaxNode) -> Option<Self>;
    pub(crate) fn syntax(&self) -> &JavaSyntaxNode;
}
```

The raw aliases can continue to exist privately inside the parser module, but
they should not be re-exported from `jolt_java_syntax`.

## Macro Schema

The macro schema should list all Java node wrappers and all grammar-family enums
in one place.

The node list provides complete wrapper coverage for Java CST node kinds:

```rust
java_cst! {
    nodes {
        ErrorNode,
        CompilationUnit,
        PackageDeclaration,
        ImportDeclaration,
        ModuleDeclaration,
        // ...
    }
}
```

Family enums encode useful Java syntax categories:

```rust
java_cst! {
    enums {
        TypeDeclaration =
            ClassDeclaration |
            RecordDeclaration |
            EnumDeclaration |
            InterfaceDeclaration |
            AnnotationInterfaceDeclaration;

        Statement =
            Block |
            EmptyStatement |
            LabeledStatement |
            ExpressionStatement |
            IfStatement |
            AssertStatement |
            SwitchStatement |
            WhileStatement |
            DoStatement |
            ForStatement |
            BreakStatement |
            YieldStatement |
            ContinueStatement |
            ReturnStatement |
            ThrowStatement |
            SynchronizedStatement |
            TryStatement |
            TryWithResourcesStatement;

        Expression =
            LiteralExpression |
            NameExpression |
            ThisExpression |
            SuperExpression |
            ParenthesizedExpression |
            ClassLiteralExpression |
            FieldAccessExpression |
            ArrayAccessExpression |
            MethodInvocationExpression |
            MethodReferenceExpression |
            ObjectCreationExpression |
            ArrayCreationExpression |
            AssignmentExpression |
            ConditionalExpression |
            InstanceofExpression |
            BinaryExpression |
            UnaryExpression |
            PostfixExpression |
            CastExpression |
            LambdaExpression |
            SwitchExpression;
    }
}
```

The schema should be explicit. The macro should not infer grammar categories
from naming conventions.

## Generated Boilerplate

The macro should generate:

- one wrapper struct per node kind,
- a sealed internal trait for node wrappers,
- `cast` and `can_cast` implementations,
- `Clone`, `Debug`, `Eq`, and `PartialEq` where appropriate,
- text range and syntax kind accessors,
- family enum casts,
- `From<ConcreteNode> for FamilyEnum` implementations,
- shared child traversal helpers used by handwritten accessors.

The generated API should be boring and uniform. Handwritten code should express
Java-specific syntax relationships, not wrapper plumbing.

## Handwritten Accessors

After complete wrapper coverage exists, add handwritten accessors for syntax
relationships that callers should not rediscover manually.

Examples:

- `CompilationUnit::package_declaration`
- `CompilationUnit::imports`
- `CompilationUnit::module_declaration`
- `CompilationUnit::type_declarations`
- `ImportDeclaration::name`
- `ClassDeclaration::name`
- `ClassDeclaration::body`
- `MethodDeclaration::name`
- `MethodDeclaration::parameters`
- `Block::statements`
- `IfStatement::condition`
- `IfStatement::then_statement`
- `IfStatement::else_statement`
- `MethodInvocationExpression::arguments`
- `ArrayType::dimensions`
- `Annotation::arguments`

Accessors should return typed wrappers or tokens. Avoid convenience APIs that do
not encode real syntax structure needed by current or imminent callers.

## Error and Recovery Policy

Recovered trees may contain `ErrorNode` wrappers, but initial formatter entry
points should still reject recovered syntax outcomes before layout.

Do not add missing-token abstractions, skipped-region wrappers, or
partial-formatting APIs in this milestone. Those belong to a later
partial-formatting policy.

## Test Plan

Add tests that fail when the wrapper layer becomes incomplete or inconsistent:

- every Java node kind is represented by exactly one concrete wrapper or an
  explicit internal exception,
- every wrapper casts only the matching `JavaSyntaxKind`,
- every family enum casts exactly the listed variants,
- every concrete family member converts into its family enum,
- parser roots from fixture corpora cast to `CompilationUnit`,
- representative fixture trees can be traversed using wrappers only,
- public API tests do not import raw `JavaSyntaxNode`, `JavaSyntaxToken`, or
  `JavaSyntaxElement`.

Avoid tests that merely duplicate enum defaults or simple accessors without
checking the wrapper contract.

## Implementation Steps

1. Add an internal `nodes` module to `jolt_java_syntax` and re-export the typed
   public syntax API from the crate root.
2. Move raw Java syntax aliases out of the crate's public exports.
3. Add the macro schema with complete node coverage.
4. Generate concrete wrapper structs and casts.
5. Generate grammar-family enums and conversions.
6. Change `JavaParse::syntax` or add a replacement method so callers receive
   `Option<CompilationUnit>`.
7. Port existing Java syntax tests and fixture summaries to wrapper traversal.
8. Add wrapper coverage tests.
9. Add handwritten syntax accessors layer by layer until the full Java CST has
   production-grade typed access.
10. Update formatter architecture notes once this milestone is implemented.

## Completion Criteria

Milestone 6 is complete when:

- `jolt_java_syntax` exposes typed CST wrappers as its public syntax API,
- raw Java red node traversal is not part of the public API,
- every Java CST node kind has wrapper coverage or an explicit documented
  exception,
- grammar-family enums cover the major Java CST categories,
- corpus fixture traversal works through wrappers,
- wrapper coverage tests fail on missing node kinds,
- no formatter layout code depends on raw Java CST traversal.
